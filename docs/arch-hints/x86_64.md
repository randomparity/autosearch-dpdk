# x86_64 Performance Optimization Checklist

## Native Optimization Reference for Intel Sapphire Rapids / AMD Zen 4+

This checklist identifies the major architectural differences between Intel and AMD
x86_64 platforms that affect DPDK performance, and provides an iterative plan for
addressing each area. Items are ordered roughly by impact and ease of diagnosis —
start at Phase 1 and work down.

---

## Architectural Comparison at a Glance

| Property | Intel (Sapphire/Emerald Rapids) | AMD (Zen 4 / Zen 5) |
|---|---|---|
| Microarchitecture | Golden Cove / Raptor Cove | Zen 4 (Genoa) / Zen 5 (Turin) |
| L1D cache line size | 64 bytes | 64 bytes |
| L1D size / core | 48 KB (12-way) | 32 KB (8-way) |
| L2 size / core | 2 MB (16-way) | 1 MB (8-way) |
| L3 cache | ~1.875 MB/core shared (up to 112.5 MB) | 32 MB per CCX (shared by 8 cores) |
| SIMD widths | SSE4.2, AVX2, AVX-512 (2x 512-bit FMA) | SSE4.2, AVX2, AVX-512 (2x 256-bit) |
| AVX-512 throttling | Yes — freq drops on heavy AVX-512 | No — no penalty on Zen 4+ |
| DDIO (NIC-to-LLC) | Yes — DCA/DDIO writes to LLC directly | No — NIC DMA goes to DRAM |
| NUMA topology | Monolithic die or SNC (sub-NUMA clusters) | CCX/CCD boundaries; Infinity Fabric |
| Memory channels / socket | 8 (DDR5) | 12 (DDR5, Zen 4 EPYC) |
| Memory ordering | TSO (Total Store Order) | TSO (Total Store Order) |
| SMT | 2 threads/core | 2 threads/core |
| Prefetchers | L1 stream, L1 stride, L2 stream, AMP | L1 stream, L2 stream, aggressive |
| RDT / CAT (LLC partitioning) | Yes — Intel RDT, MBA | No equivalent |

## Phase 1 — Compiler & Build System (Day 1)

These items produce immediate gains with minimal code changes.

### 1.1 Set `-march=` and `-mtune=` correctly

The single highest-impact compiler change. Incorrect or missing arch flags leave
performance on the table by disabling ISA extensions.

- [ ] Identify current `-march` / `-mtune` in Meson cross-files, `meson_options.txt`,
  and DPDK's `config/` directory
- [ ] For Intel Sapphire Rapids: `-march=sapphirerapids` (GCC 12+, Clang 14+)
- [ ] For AMD Zen 4 (Genoa): `-march=znver4` (GCC 13+, Clang 16+)
- [ ] For AMD Zen 5 (Turin): `-march=znver5` (GCC 14+, Clang 18+)
- [ ] For portable builds: `-march=x86-64-v3` (AVX2 + FMA + BMI2) or
  `-march=x86-64-v4` (adds AVX-512)
- [ ] For build-machine-native: `-march=native -mtune=native`
- [ ] Verify DPDK's `machine` meson option: `meson configure -Dmachine=native`
  or `meson configure -Dmachine=sapphirerapids`
- [ ] Check GCC version >= 12 (Sapphire Rapids) or >= 13 (Zen 4 support)

### 1.2 Enable link-time optimization (LTO) and profile-guided optimization (PGO)

- [ ] Add `-flto` to CFLAGS/LDFLAGS (DPDK: `-Db_lto=true` in meson)
- [ ] Run a representative workload and generate PGO profile data with
  `-fprofile-generate`
- [ ] Rebuild with `-fprofile-use` against collected profiles
- [ ] Verify LTO does not regress build times unacceptably for development cycles

### 1.3 AVX-512 frequency throttling awareness (Intel only)

Intel CPUs reduce core frequency during sustained AVX-512 workloads. This can
hurt surrounding non-SIMD code.

- [ ] Measure frequency with `turbostat` during AVX-512-heavy workloads
- [ ] On Intel: consider `-mprefer-vector-width=256` to avoid 512-bit throttling
  in mixed workloads
- [ ] On AMD Zen 4+: no throttling — AVX-512 is safe to use without penalty
- [ ] DPDK PMDs (ice, i40e) have both scalar and vector paths — profile which
  performs better on your specific Intel SKU
- [ ] Intel Granite Rapids (next-gen) reduces the throttling penalty significantly

### 1.4 GCC vs Clang for DPDK

- [ ] GCC typically generates better AVX-512 code for DPDK on Intel
- [ ] Clang sometimes produces tighter scalar code and better inlining decisions
- [ ] Both support `-march=native` for auto-detection — verify with
  `gcc -march=native -Q --help=target | grep march`
- [ ] Profile both compilers on your specific workload before committing

## Phase 2 — Memory Subsystem & Page Size (Week 1)

Memory access patterns dominate DPDK performance.

### 2.1 Hugepage configuration

DPDK requires hugepages. Correct sizing eliminates TLB misses from the hot path.

- [ ] Enable 1 GB hugepages via kernel cmdline:
  `hugepagesz=1G hugepages=N default_hugepagesz=1G`
- [ ] For smaller allocations: 2 MB hugepages via
  `echo N > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages`
- [ ] Mount hugetlbfs: `mount -t hugetlbfs nodev /dev/hugepages`
- [ ] Use 1 GB pages for DPDK mempools — eliminates TLB misses entirely for
  pools up to 1 GB
- [ ] Verify with `perf stat -e dTLB-load-misses` — should be near zero
- [ ] Disable transparent huge pages (THP) for DPDK workloads:
  `echo never > /sys/kernel/mm/transparent_hugepage/enabled`

### 2.2 DDIO — Data Direct I/O (Intel only)

DDIO allows NICs to DMA directly into LLC, bypassing DRAM. Critical for latency.

- [ ] Verify DDIO is enabled: check BIOS setting "Intel Data Direct I/O"
- [ ] DDIO uses 2 LLC ways by default — may be insufficient for high packet rates
- [ ] Tune DDIO allocation via MSR `0xC8B` (IIO LLC ways) if packet buffers
  evict working set from LLC
- [ ] On AMD: no DDIO — NIC DMA goes to DRAM. Prefetching matters more;
  ensure `rte_prefetch0()` is used in RX paths
- [ ] Monitor LLC miss rate: `perf stat -e LLC-load-misses,LLC-store-misses`

### 2.3 NUMA topology

Multi-socket and sub-NUMA clustering require careful memory and core placement.

- [ ] Map topology: `numactl --hardware`, `lstopo` (hwloc)
- [ ] Intel SNC (Sub-NUMA Clustering): splits LLC into 2 or 4 NUMA domains
  per socket — verify with `numactl -H` and align DPDK `--socket-mem`
- [ ] AMD CCD/CCX: each CCX (8 cores) shares 32 MB L3. Cross-CCX access
  adds ~20 ns. Pin DPDK lcores to a single CCX when possible
- [ ] Bind NIC IRQs and DPDK lcores to the same NUMA node as the NIC:
  `cat /sys/class/net/<iface>/device/numa_node`
- [ ] Use `--socket-mem` or `--huge-dir` per NUMA node in EAL args
- [ ] For AMD EPYC: use `NPS=1` BIOS setting for uniform memory unless
  workload benefits from partitioned NUMA (NPS=2/4)

## Phase 3 — SIMD & Vectorization (Week 2)

### 3.1 ISA level selection for DPDK

DPDK selects vector code paths at build time based on `-march`. Each PMD may
have scalar, SSE, AVX2, and AVX-512 implementations.

- [ ] SSE4.2 is the minimum DPDK requirement — always available
- [ ] AVX2: enables 256-bit vector RX/TX in ice, i40e, mlx5 PMDs
- [ ] AVX-512: enables 512-bit bulk packet processing — highest throughput
  on AMD Zen 4+ and Intel (with throttling caveat)
- [ ] Verify active vector path: `grep -r "Vector Rx\|Vector Tx" dpdk/drivers/`
  or check PMD log output at startup
- [ ] DPDK `rte_memcpy` selects SSE/AVX2/AVX-512 variant at build time —
  ensure it matches your target ISA

### 3.2 Key vector PMD paths

- [ ] `ice` (Intel E810): AVX2 and AVX-512 vector RX/TX — up to 2x throughput
  vs scalar on burst workloads
- [ ] `i40e` (Intel XL710/X710): AVX2 and AVX-512 vector paths
- [ ] `mlx5` (Mellanox ConnectX): AVX2-optimized RX/TX, enhanced CQE parsing
- [ ] Vector paths require aligned mbufs and no offload flags — verify with
  `RTE_ETH_RX_OFFLOAD_*` configuration
- [ ] Use `rte_eth_dev_info_get()` to confirm vector path is active

### 3.3 Auto-vectorization tuning

- [ ] Build with `-O3 -fopt-info-vec-missed` (GCC) or
  `-Rpass-missed=loop-vectorize` (Clang) to find missed opportunities
- [ ] Add `restrict` qualifiers to pointer parameters in hot paths
- [ ] Restructure data layouts: AoS to SoA for batch processing
- [ ] Use `__builtin_expect()` for branch hints in packet classification
- [ ] Verify vectorized output: `objdump -d | grep -c 'vmovdq\|vpadd\|vpshuf'`

## Phase 4 — Memory Ordering & Concurrency (Week 2-3)

### 4.1 TSO simplifies x86 concurrency

x86 provides Total Store Order — stores from a single core are seen in order
by all other cores. This makes most barriers free in hardware.

- [ ] `rte_smp_rmb()` / `rte_smp_wmb()` compile to compiler barriers only
  (no instructions emitted) on x86 — cost is near zero
- [ ] `rte_mb()` compiles to `mfence` (~30-50 cycles) — avoid in hot paths
- [ ] `LOCK`-prefixed atomics (`lock cmpxchg`, `lock xadd`) cost ~15-30 cycles
  on uncontended lines, more under contention
- [ ] C11 `memory_order_acquire` / `memory_order_release` are free on x86 —
  use them instead of `memory_order_seq_cst` where possible
- [ ] `memory_order_seq_cst` stores emit `mfence` or `xchg` — measurable cost

### 4.2 False sharing at 64-byte cache line boundaries

- [ ] Per-lcore counters, stats, and ring pointers must be 64-byte aligned
- [ ] Use `__rte_cache_aligned` attribute on per-lcore structures
- [ ] Run `perf c2c record` + `perf c2c report` to detect false sharing
- [ ] DPDK `rte_ring`: producer and consumer heads are on separate cache lines
  by design — verify custom ring implementations do the same
- [ ] Pad independent hot fields: `char pad[RTE_CACHE_LINE_SIZE]`

### 4.3 PAUSE instruction for spin loops

The `PAUSE` instruction yields the pipeline during spin-waits. Its latency
differs significantly between vendors.

- [ ] Intel Skylake+: `PAUSE` takes ~140 cycles (was ~10 on older CPUs)
- [ ] AMD Zen 1-3: `PAUSE` takes ~65 cycles
- [ ] AMD Zen 4+: `PAUSE` takes ~16-32 cycles
- [ ] DPDK uses `rte_pause()` which maps to `_mm_pause()` — appropriate for
  both vendors
- [ ] For Intel: high PAUSE latency means spin loops converge slower — consider
  `rte_power_pause()` or event-driven alternatives for long waits

## Phase 5 — Intel vs AMD Platform Specifics (Week 3)

### 5.1 Intel-specific optimizations

- [ ] DDIO tuning: increase LLC ways for DDIO if NIC traffic evicts working set
  (MSR `0xC8B`)
- [ ] RDT / CAT: partition LLC between DPDK and OS with `resctrl` filesystem —
  prevents OS activity from evicting packet buffers
- [ ] Non-temporal stores: `_mm512_stream_si512()` bypasses cache for write-only
  buffers — saves LLC pollution on TX paths
- [ ] Intel Speed Select Technology (SST): pin high-priority cores to higher
  P-states for latency-sensitive lcores
- [ ] Sub-NUMA Clustering (SNC): disable for DPDK unless memory is carefully
  partitioned — SNC halves effective LLC per NUMA domain

### 5.2 AMD-specific optimizations

- [ ] No DDIO: NIC DMA lands in DRAM, not LLC. Software prefetching
  (`rte_prefetch0`) in RX burst functions is essential
- [ ] AVX-512 without throttling: Zen 4+ runs AVX-512 at full frequency —
  always prefer `-march=x86-64-v4` or `-march=znver4` on AMD
- [ ] CCX topology: 8 cores share 32 MB L3 within a CCX. Cross-CCX traffic
  traverses Infinity Fabric (~40-80 ns additional latency)
- [ ] Pin all DPDK lcores within a single CCX for lowest L3 latency
- [ ] Infinity Fabric clock: tied to memory clock on some SKUs. Higher DDR5
  speed improves cross-CCD bandwidth
- [ ] AMD Preferred Core: let the OS scheduler select the fastest physical
  cores, or pin DPDK lcores to known-good cores via `isolcpus`

## Phase 6 — Kernel & OS Tuning (Week 3-4)

### 6.1 CPU isolation for DPDK lcores

- [ ] `isolcpus=<lcore-list>` on kernel cmdline — removes DPDK cores from
  scheduler
- [ ] `nohz_full=<lcore-list>` — disables timer ticks on isolated cores
- [ ] `rcu_nocbs=<lcore-list>` — offloads RCU callbacks from DPDK cores
- [ ] Verify isolation: `cat /sys/devices/system/cpu/isolated`
- [ ] Combine all three for jitter-free packet processing

### 6.2 IOMMU and VFIO

- [ ] `iommu=pt` on kernel cmdline — passthrough mode for VFIO, eliminates
  IOMMU translation overhead
- [ ] `intel_iommu=on iommu=pt` (Intel) or `amd_iommu=on iommu=pt` (AMD)
- [ ] Bind NICs to `vfio-pci` driver: `dpdk-devbind.py -b vfio-pci <PCI>`

### 6.3 IRQ affinity and power management

- [ ] Pin NIC IRQs away from DPDK lcores:
  `echo <cpu-mask> > /proc/irq/<N>/smp_affinity`
- [ ] Set CPU governor to performance:
  `cpupower frequency-set -g performance`
- [ ] Disable C-states deeper than C1:
  `intel_idle.max_cstate=1` or `processor.max_cstate=1`
- [ ] Disable P-state scaling for consistent frequency:
  `intel_pstate=disable` and use `acpi-cpufreq` governor

### 6.4 Transparent hugepages

- [ ] Disable THP for DPDK: `echo never > /sys/kernel/mm/transparent_hugepage/enabled`
- [ ] THP causes non-deterministic latency spikes from `khugepaged` compaction
- [ ] DPDK uses explicit hugetlbfs — THP provides no benefit and adds jitter

## Phase 7 — Profiling & Validation (Ongoing)

### 7.1 Profiling tools

- [ ] `perf record -g` + `perf report` — basic CPU profiling
- [ ] `perf stat -d` — hardware counter summary (IPC, cache misses, branch
  mispredictions)
- [ ] `perf c2c record` + `perf c2c report` — false sharing detection
- [ ] Intel VTune: Top-Down Microarchitecture Analysis (TMAM) for pipeline
  bottleneck classification (frontend/backend/speculation/retiring)
- [ ] `toplev.py` (pmu-tools): command-line TMAM analysis without VTune
- [ ] `perf stat -e topdown-retiring,topdown-bad-spec,topdown-fe-bound,topdown-be-bound`
  (Linux 5.13+ with perf_event TMAM support)
- [ ] For DPDK: `dpdk-proc-info -- --stats` for per-port/per-queue counters
- [ ] `testpmd` with `--stats-period 2` for live throughput monitoring

### 7.2 Regression testing matrix

- [ ] Benchmark at multiple burst sizes (1, 4, 16, 32, 64 packets)
- [ ] Test with and without AVX-512 (`-mprefer-vector-width=256` vs `512`)
- [ ] Verify NUMA-local vs NUMA-remote throughput (should see 10-30% delta)
- [ ] Test with `isolcpus` and without — quantify jitter reduction
- [ ] Monitor `turbostat` for frequency stability during sustained traffic

|---|---|---|
| Cache line | 64 B | 64 B |
| Base page | 4 KB | 4 KB |
| Huge pages | 2 MB, 1 GB | 2 MB, 1 GB |
| L1D latency | ~5 cycles | ~4 cycles |
| L1D size / core | 48 KB | 32 KB |
| L2 latency | ~12-14 cycles | ~12 cycles |
| L2 size / core | 2 MB | 1 MB |
| L3 latency | ~40-50 cycles | ~40 cycles (same CCX) |
| L3 size / core | ~1.875 MB (shared) | ~4 MB (32 MB / 8 cores) |

| `PAUSE` latency | ~140 cycles | ~16-32 cycles |
| DDIO | Yes (2 LLC ways default) | No |
| LLC partitioning (RDT/CAT) | Yes | No |

## Appendix: Compiler Flag Quick Reference

```bash
# Intel Sapphire Rapids optimized build
CFLAGS="-O3 -march=sapphirerapids -mtune=sapphirerapids -flto"

# AMD Zen 4 (Genoa) optimized build
CFLAGS="-O3 -march=znver4 -mtune=znver4 -flto"

# AMD Zen 5 (Turin) optimized build (GCC 14+, Clang 18+)
CFLAGS="-O3 -march=znver5 -mtune=znver5 -flto"

# Portable x86-64-v3 (AVX2 + FMA + BMI2 — broad compatibility)
CFLAGS="-O3 -march=x86-64-v3 -mtune=generic -flto"

# Portable x86-64-v4 (adds AVX-512 — Zen 4+ and Intel SPR+)
CFLAGS="-O3 -march=x86-64-v4 -mtune=generic -flto"

# Intel with AVX-512 throttling mitigation
CFLAGS="-O3 -march=sapphirerapids -mprefer-vector-width=256 -flto"

# DPDK meson build (native auto-detect)
meson setup build -Dmachine=native -Db_lto=true -Dbuildtype=release

# Diagnostic flags
CFLAGS+=" -fopt-info-vec-missed"     # Report failed vectorization (GCC)
CFLAGS+=" -Rpass-missed=loop-vectorize"  # Report failed vectorization (Clang)
```
