# s390x Performance Optimization Checklist

## Porting from amd64 to z/Architecture (z15 / z16 / z17)

This checklist identifies the major architectural differences between amd64
(x86_64) and s390x that affect application performance, and provides an
iterative plan for addressing each area. Items are ordered roughly by impact
and ease of diagnosis — start at Phase 1 and work down.

---

## Architectural Comparison at a Glance

| Property | amd64 (x86_64) | s390x (z15/z16/z17) |
|---|---|---|
| Default page size | 4 KB | 4 KB |
| Huge page sizes | 2 MB, 1 GB | 1 MB, 2 GB |
| L1D cache line size | 64 bytes | 256 bytes |
| L2 cache line size | 64 bytes | 256 bytes |
| L2 cache per core | 256 KB-2 MB (varies) | 4 MB I+D (z15), 32 MB unified (z16), 36 MB unified (z17) |
| L3 cache | Shared, 1-4 MB/core | 256 MB shared/drawer (z15); virtual victim cache (z16/z17) |
| SIMD width | 128-bit SSE, 256-bit AVX2, 512-bit AVX-512 | 128-bit Vector Extension (VXE) |
| Matrix acceleration | AMX (Sapphire Rapids+) | None |
| Memory ordering | TSO (strong — stores are totally ordered) | Strong (similar to TSO) |
| Endianness | Little-endian | Big-endian |
| SMT depth | 2 threads/core (typical) | 2 threads/core (z15+) |
| Long double format | 80-bit x87 extended | IEEE 128-bit binary128 |
| Instruction encoding | Variable-length (1-15 bytes) | Variable-length (2, 4, or 6 bytes) |
| ABI stack linkage | 128-byte red zone | 160-byte register save area |

## Phase 1 — Compiler & Build System (Day 1)

These items produce immediate gains with minimal code changes.

### 1.1 Set `-march=` correctly

The single highest-impact compiler change. Many s390x builds default to
`-march=z196` or older, missing substantial ISA improvements in z15+.

- [ ] Identify current `-march` in all Makefiles, CMake, Meson, spec files
- [ ] Set `-march=z15` minimum for z15 targets (enables VXE2, DFLTCC)
- [ ] Set `-march=z16` for z16 targets (SIMD enhancements, NNPA)
- [ ] Set `-march=z17` for z17 targets (GCC 15+, further vector enhancements)
- [ ] Verify GCC version >= 10 for z15, >= 13 for z16, >= 15 for z17
- [ ] Verify LLVM/Clang version >= 11 for z15, >= 16 for z16
- [ ] Add `-mvx` explicitly if not implied by `-march` to enable vector extension
- [ ] Check for stale `-march=z196` or `-march=zEC12` generic fallbacks

### 1.2 Enable link-time optimization (LTO) and profile-guided optimization

- [ ] Add `-flto` to CFLAGS/LDFLAGS
- [ ] Run a representative workload and generate PGO profile data with
      `-fprofile-generate`
- [ ] Rebuild with `-fprofile-use` against collected profiles
- [ ] Verify LTO doesn't regress build times unacceptably for dev cycles

### 1.3 Verify long double format

s390x uses IEEE 128-bit long double (`binary128`) on modern distros.
Older builds may use IBM hex floating-point format. Mixing the two in
shared libraries causes silent corruption.

- [ ] Check `LDBL_MANT_DIG` -- 113 = IEEE binary128, 56 = IBM hex float
- [ ] Ensure all linked libraries (OpenSSL, glibc, libstdc++) use the
      same format
- [ ] If building with Clang, verify the long double format matches the
      system default

### 1.4 Audit third-party / vendored libraries

- [ ] Check whether OpenBLAS, FFTW, Eigen, etc. are built with z15-aware
      settings (not falling back to generic s390x)
- [ ] Verify `glibc` version >= 2.28 for z15 hwcap support; >= 2.31 for
      optimized string/mem routines using VXE
- [ ] Check that any JIT engines (LuaJIT, V8, SpiderMonkey) have s390x
      backends enabled

## Phase 2 — Memory Subsystem & Page Size (Week 1)

The 256-byte cache line is the #1 porting issue for s390x.

### 2.1 Cache line size (256 bytes vs. 64 bytes)

s390x cache lines are 4x the size of x86 -- the largest of any major
server architecture. This profoundly affects data structure layout,
false sharing, and prefetch behavior.

- [ ] Grep for `__cacheline_aligned`, `__attribute__((aligned(64)))`, or
      `#define CACHE_LINE_SIZE 64`
- [ ] Update all cache-line alignment macros to 256 bytes for s390x
- [ ] Audit lock-free data structures: per-thread counters, ring buffers,
      work-stealing queues -- pad to 256 bytes between independent fields
- [ ] In DPDK: verify `RTE_CACHE_LINE_SIZE` is 256 on your build (check
      `config/s390x/` or `lib/eal/s390x/`)
- [ ] Review struct layout in hot paths -- four "independent" fields that
      each occupy 64 bytes on x86 may share one 256-byte cache line and
      contend on s390x
- [ ] Ring buffers and descriptor arrays: ensure producer/consumer metadata
      is separated by >= 256 bytes
- [ ] Profile false sharing with `perf c2c record -a` then `perf c2c report`

### 2.2 Page size awareness

s390x Linux uses 4 KB base pages -- the same as x86. No page size
surprise here (unlike ppc64le with its 64 KB pages).

- [ ] Confirm with `getconf PAGESIZE` (should return 4096)
- [ ] Existing 4 KB page assumptions carry over from x86 without changes
- [ ] Custom allocators should still use `sysconf(_SC_PAGESIZE)` rather
      than hardcoded values for portability

### 2.3 Huge pages

- [ ] Enable 1 MB huge pages: `echo N > /proc/sys/vm/nr_hugepages`
- [ ] For large buffers, use `mmap()` with `MAP_HUGETLB` and appropriate
      size flags for 1 MB or 2 GB pages
- [ ] Enable transparent huge pages if appropriate:
      `echo always > /sys/kernel/mm/transparent_hugepage/enabled`
- [ ] Profile TLB miss rate: `perf stat -e dTLB-load-misses,iTLB-load-misses`
- [ ] For DPDK: configure `--huge-dir` and EAL huge page parameters.
      Note: x86 uses 2 MB huge pages by default; s390x uses 1 MB

## Phase 3 — SIMD & Vectorization (Week 2)

### 3.1 Migrate x86 SIMD intrinsics

There is no hardware translation from SSE/AVX to VXE. All x86
intrinsics must be rewritten.

- [ ] Inventory all `#include <immintrin.h>`, `<xmmintrin.h>`,
      `<emmintrin.h>` usage
- [ ] Evaluate porting strategy per hot path:
  - **Option A**: Use compiler auto-vectorization (`-O3 -ftree-vectorize`)
    and verify with `-fopt-info-vec-all`
  - **Option B**: Use `<vecintrin.h>` (s390x vector intrinsics) directly
    for critical paths
  - **Option C**: Use a portability layer like `simde` (SIMDe) or Highway
    (`google/highway`)
- [ ] Replace `_mm_prefetch()` with `__builtin_prefetch()`
- [ ] Note: VXE registers are 128-bit (like SSE), so 256-bit AVX2 code
      needs to be split into two VXE operations
- [ ] For AVX-512 code: significant restructuring needed -- no equivalent
- [ ] In DPDK: check for `RTE_ARCH_S390X` guards in SIMD fast paths

### 3.2 Leverage VXE2 (z15+) and NNPA (z16+)

VXE2 adds enhanced string operations and IEEE binary128 in vector
registers. NNPA (z16+) provides neural network processing assists.

- [ ] Use VXE2 string builtins for pattern matching and comparison in
      packet classification
- [ ] z16+ NNPA: 128-bit vector operations optimized for AI inference
      (INT8, FP16, BFloat16)
- [ ] For z16+: SIMD-enhanced sort and search instructions

### 3.3 Verify auto-vectorization effectiveness

- [ ] Build with `-O3 -fopt-info-vec-missed` and review which loops
      failed to vectorize
- [ ] Common blocker: pointer aliasing -- add `restrict` qualifiers
- [ ] Common blocker: non-contiguous memory -- restructure data layouts
      (AoS to SoA)
- [ ] Use `-mvx -mvxe` explicitly if not implied by `-march`
- [ ] Check `objdump -d` for vector instructions in hot functions
      (`vl`, `vst`, `vfa`, `vfm`, `vperm`)

## Phase 4 — Memory Ordering & Concurrency (Week 2-3)

### 4.1 Audit memory ordering assumptions

s390x has a **strong memory model** -- similar to x86 TSO. Stores are
ordered with respect to other stores, and loads are ordered with respect
to other loads. This means most lock-free code that works on x86 will
also work on s390x without additional barriers.

- [ ] Existing x86 lock-free algorithms generally work correctly on s390x
      -- the memory model is comparably strong
- [ ] `BCR 15,0` is the full memory barrier (serialization), but it is
      rarely needed due to strong ordering
- [ ] Compare-and-Swap is native: `CS` (32-bit) and `CSG` (64-bit)
      instructions -- no need for LL/SC emulation
- [ ] Unlike POWER (weak ordering), you do NOT need to add extra barriers
      when porting from x86 to s390x
- [ ] Still use C11/C++11 atomics (`<stdatomic.h>`, `<atomic>`) for
      correctness and portability -- the compiler will elide unnecessary
      barriers on s390x
- [ ] Run `ThreadSanitizer` on s390x to catch any latent races

### 4.2 NUMA topology

Modern z/Architecture systems have NUMA characteristics at the drawer
and book level.

- [ ] Map topology with `numactl --hardware` or `lscpu`
- [ ] Bind processes to a single NUMA node: `numactl --cpunodebind=0
      --membind=0`
- [ ] For DPDK: ensure EAL arguments include `--socket-mem` and core
      pinning within a single NUMA node
- [ ] Profile NUMA effects: `perf stat -e numa-*` events

## Phase 5 — Endianness (Week 2-3)

### 5.1 Big-endian byte order

s390x is **big-endian** -- this is the major software porting challenge.
However, big-endian is network byte order, which provides a real
advantage for packet processing.

- [ ] `ntohl()` / `ntohs()` / `htonl()` / `htons()` are **no-ops** on
      s390x -- network byte order is native byte order
- [ ] This means packet header parsing is faster: no byte-swap overhead
      for protocol fields (Ethernet type, IP addresses, TCP ports)
- [ ] DPDK `rte_byteorder.h` handles most conversions automatically --
      verify it detects s390x correctly

### 5.2 Audit little-endian assumptions

Code that works on x86 often has hidden little-endian assumptions that
silently produce wrong results on s390x.

- [ ] Audit all `*(uint32_t *)ptr` and `*(uint16_t *)ptr` casts on byte
      buffers -- these produce reversed values on big-endian
- [ ] Check bitfield structs: C bitfield layout order is
      implementation-defined and typically reverses between LE and BE
- [ ] Audit union type-punning (e.g., `union { uint32_t u32; uint8_t
      bytes[4]; }`) -- byte order within the union differs
- [ ] Check serialization/deserialization code: any protocol that
      specifies little-endian wire format needs explicit conversion
- [ ] Audit shift-and-mask patterns like `(val >> 24) & 0xFF` -- these
      extract different bytes depending on endianness context
- [ ] Search for `#if __BYTE_ORDER__` guards -- ensure s390x (big-endian)
      paths are correct and tested
- [ ] In DPDK: verify `RTE_BYTE_ORDER` is `RTE_BIG_ENDIAN` on s390x
- [ ] Run the full test suite on s390x hardware -- many endianness bugs
      only manifest at runtime

## Phase 6 — Hardware Acceleration (Week 3-4)

### 6.1 CPACF -- cryptographic acceleration

The Central Processor Assist for Cryptographic Functions provides inline
hardware acceleration without needing a coprocessor.

- [ ] Verify OpenSSL uses CPACF: `openssl speed -evp aes-256-gcm` should
      show high throughput
- [ ] CPACF supports: AES (128/192/256), SHA-1, SHA-256, SHA-512,
      SHA-3, GHASH, PRNG
- [ ] For DPDK crypto: check if `dpdk-crypto-s390x` PMD or OpenSSL PMD
      leverages CPACF
- [ ] z15+: AES-GCM fully in hardware with very low latency

### 6.2 DFLTCC -- hardware deflate compression

z15+ provides hardware-accelerated deflate compression/decompression
via the DFLTCC instruction.

- [ ] For compressed packet payloads: use zlib-ng or glibc's zlib with
      DFLTCC support enabled (`--with-dfltcc`)
- [ ] Throughput: DFLTCC achieves 10-50 GB/s depending on data pattern
- [ ] Verify kernel support: `grep DFLTCC /proc/cpuinfo`

### 6.3 CRC and checksum acceleration

- [ ] s390x provides SIMD-accelerated CRC32 via vector instructions
- [ ] Use `__builtin_s390_*` intrinsics or let the compiler auto-optimize
      CRC loops with `-mvx`
- [ ] For DPDK packet checksums: verify that CRC offload or software CRC
      uses vector instructions on s390x

## Phase 7 — Profiling & Validation (Ongoing)

### 7.1 Profiling tools

- [ ] `perf record -g` + `perf report` -- basic CPU profiling
- [ ] `perf stat -d` -- hardware counter summary (IPC, cache misses,
      branch mispredictions)
- [ ] `perf c2c` -- false sharing analysis (critical for 256-byte cache
      lines)
- [ ] `perf stat -e dTLB-load-misses,iTLB-load-misses` -- TLB pressure
- [ ] `objdump -d binary | grep -c 'bcr\s*15'` -- barrier frequency
- [ ] `perf annotate` -- per-instruction cycle attribution
- [ ] For DPDK: use `dpdk-test-flow-perf`, `dpdk-testpmd` with
      `--stats-period`

### 7.2 Regression testing matrix

- [ ] Endianness: run full test suite on s390x hardware -- this is the
      most important validation step
- [ ] Correctness: run under TSan on s390x (memory model is strong, but
      still catches data races)
- [ ] Memory: run under Valgrind or ASan to catch alignment issues with
      256-byte cache lines
- [ ] Performance: benchmark at SMT1 and SMT2
- [ ] NUMA: test with memory bound to local vs. remote node
- [ ] QEMU user-mode: use `qemu-s390x` for quick functional testing of
      endianness on x86 development machines

## Quick Reference: Key Numbers

| Metric | x86_64 | z15 | z16 | z17 |
|---|---|---|---|---|
| Cache line | 64 B | 256 B | 256 B | 256 B |
| Base page | 4 KB | 4 KB | 4 KB | 4 KB |
| Huge pages | 2 MB, 1 GB | 1 MB, 2 GB | 1 MB, 2 GB | 1 MB, 2 GB |
| L1D latency | 4-5 cycles | ~4 cycles | ~4 cycles | ~4 cycles |
| L2 latency | ~12 cycles | ~12 cycles | ~12 cycles | ~12 cycles |
| L2 size/core | 256 KB-2 MB | 4 MB | 4 MB | 4 MB |
| L3 cache | 1-4 MB/core | 256 MB/drawer | 256 MB/drawer | 256 MB/drawer |
| Max SMT | 2 | 2 | 2 | 2 |
| Vector width | SSE=128, AVX=256 | 128-bit VXE2 | 128-bit VXE2+ | 128-bit VXE2+ |
| DRAM bandwidth/socket | ~100-200 GB/s | ~320 GB/s | ~400+ GB/s | ~400+ GB/s |
| `BCR 15,0` cost | N/A (`mfence` ~30-50 cy) | ~20-40 cycles | ~20-40 cycles | ~20-40 cycles |
| Endianness | Little | Big | Big | Big |
| Crypto accel | AES-NI | CPACF inline | CPACF inline | CPACF inline |
| Compression HW | None (QAT optional) | DFLTCC | DFLTCC | DFLTCC |

## Appendix: Compiler Flag Quick Reference

```bash
# z15 optimized build (VXE2, DFLTCC)
CFLAGS="-O3 -march=z15 -mvx -mzvector -flto"

# z16 optimized build (SIMD enhancements, NNPA)
CFLAGS="-O3 -march=z16 -mvx -mzvector -flto"

# z17 optimized build (GCC 15+)
CFLAGS="-O3 -march=z17 -mvx -mzvector -flto"

# Diagnostic flags
CFLAGS+=" -fopt-info-vec-missed"     # Report failed vectorization
CFLAGS+=" -fopt-info-loop-all"       # Loop optimization report
```
